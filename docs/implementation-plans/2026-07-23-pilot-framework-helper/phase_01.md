# Pilot Framework Helper (`@arcjet/ai`) Implementation Plan — Phase 1: Package scaffolding

**Goal:** `@arcjet/ai` exists as a buildable, publishable workspace package wired into the release pipeline.

**Architecture:** New workspace package `arcjet-ai/` in the arcjet-js npm-workspaces monorepo, scaffolded on the conventions of `sensitive-info-rampart/` (the most recently added package, July 2026): ESM-only, `tsdown` build, `tsgo --noEmit` typecheck, Node built-in test runner, tests in `test/*.test.ts` importing from built `dist/`. Published at dependency level 4 (framework integrations) alongside `@arcjet/next`.

**Tech Stack:** TypeScript 6.0.3, tsdown 0.22.3, tsgo (`@typescript/native-preview`), node:test, npm workspaces + turbo, release-please.

**Scope:** Phase 1 of 6 from `docs/design-plans/2026-07-23-pilot-framework-helper.md` (design Phases 7–8 execute in the separate `review` repository and are not part of this plan).

**Codebase verified:** 2026-07-23 (codebase-investigator + direct file reads; npm registry checked directly)

## Verified facts this phase relies on

- Root `package.json` has `"workspaces": ["*"]` — a top-level `arcjet-ai/` directory is picked up automatically; no root edits needed.
- Template package: `sensitive-info-rampart/` (package.json shape, `tsconfig.json` extending `../tsconfig.base.json`, simple `tsdown.config.ts`, per-package Apache-2.0 LICENSE copy).
- Peer-dependency convention for framework adapters: broad ranges (`next: ">=13"` in `arcjet-next/package.json`, `fastify: ">=5"` in `arcjet-fastify/package.json`). Per the design, `ai` gets `">=7 <8"` (deliberately capped; widened deliberately later).
- Sibling peer deps use exact lockstep versions (`sensitive-info-rampart` peer-deps `arcjet: "1.9.1"`); release-please's `node-workspace` plugin has `updatePeerDependencies: true`, so `@arcjet/guard: "1.9.1"` stays in lockstep automatically.
- Current monorepo version: **1.9.1** (`.github/.release-please-manifest.json`).
- Registry facts (checked 2026-07-23 via `npm view`): `ai` latest = **7.0.36** (engines `node >=22`, peer dep `zod ^3.25.76 || ^4.1.8`); `zod` latest 4.x = **4.4.3**.
- `CHANGELOG.md` is generated by release-please at first release — do NOT create one.
- CONTRIBUTING.md "Adding a new package" (lines 135–156) requires: manifest entry, config `packages` entry + `linked-versions` component, `publish.yml` workspace flag at the right level, and notes the first publish is manual (operational step after merge — not part of this plan).
- Lint/format run from the root (`npm run lint` → oxlint, `npm run format:check` → oxfmt); no per-package lint script needed.

Notes for the implementor (do not put these in the file):
- `zod` is a devDependency only because `ai`'s published typings reference zod types; `@arcjet/ai` itself must never import zod at runtime (later phases use `jsonSchema()` from `ai` instead). `ai` declares `zod` as a peer (`^3.25.76 || ^4.1.8`), so without it `npm install` warns and `tsgo` cannot resolve `ai`'s types.
- Both peer deps are required (no `peerDependenciesMeta.optional`), unlike rampart's optional peers: the package is unusable without both `ai` and `@arcjet/guard`.
- devDependency versions are exact, matching repo convention (compare `sensitive-info-rampart/package.json`).


**Step 2: Add the release-please config entry**

In `.github/release-please-config.json`:

a. In the `"packages"` object, add alphabetically (after `"arcjet"`, before `"arcjet-astro"`):


**Step 3: Add the publish workflow entry**

In `.github/workflows/publish.yml`, find the block commented `# Level 4: public framework integrations` (the `npm publish` list starting with `--workspace @arcjet/astro` and ending with `--workspace @arcjet/sveltekit`) and add `@arcjet/ai` first in its alphabetical list:

```yaml
          # Level 4: public framework integrations
          npm publish --tag "$DIST_TAG" \
            --workspace @arcjet/ai \
            --workspace @arcjet/astro \
```

(`@arcjet/ai` depends only on `@arcjet/guard`, published at Level 3, so Level 4 is correct. Anchor the edit on the `--workspace @arcjet/astro` line rather than a line number.)
